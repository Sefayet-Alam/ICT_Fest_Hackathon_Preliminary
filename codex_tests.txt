CoWork Booking API - Critical Manual Endpoint Tests
===================================================

Source files read: HANDOFF_FOR_CODEX.md, README.md, bug_report.md, BUG_LEDGER.md, TESTCASES.md, TESTCASES.json, HACKATHON_STATE.md.
No application code changes are needed for these tests.

How to run manually in Chrome
-----------------------------
1. Start Docker, then open: http://localhost:8000/docs
2. Use Swagger "Try it out" for POST/GET requests.
3. For protected endpoints, click "Authorize" and enter: Bearer <access_token>
4. Use a unique run suffix to avoid old persisted data:
   RUN = any short value, e.g. 20260709a
   ORG_A = codex-acme-RUN
   ORG_B = codex-other-RUN
5. Capture these values as you test:
   ADMIN_A_ACCESS, ADMIN_A_REFRESH, MEMBER_A_ACCESS, MEMBER_B_ACCESS,
   ROOM_A_ID, ROOM_B_ID, BOOKING_A_ID, MEMBER_BOOKING_ID

Common expected error shape
---------------------------
Application errors should be JSON:
  {"detail": "<message>", "code": "<CODE>"}

FastAPI request validation errors return HTTP 422 with the default "detail" array.
Missing, invalid, expired, blacklisted, or wrong-type tokens should return HTTP 401.

8. POST /bookings
-----------------

Use ROOM_A_ID. Dates below are future relative to 2026-07-09.
If you test after these dates, replace them with future UTC datetimes.

T-PRICE - price = hourly_rate_cents x whole hours
Authorize with MEMBER_A_ACCESS.
Body:
  {"room_id":ROOM_A_ID,"start_time":"2026-12-01T09:00:00Z","end_time":"2026-12-01T11:00:00Z"}
Expected:
  HTTP 201
  JSON has status:"confirmed", price_cents:2000
Save id as MEMBER_BOOKING_ID.

T-TZ-OFFSET - offset input converted to UTC
Body:
  {"room_id":ROOM_A_ID,"start_time":"2026-12-02T12:00:00+06:00","end_time":"2026-12-02T13:00:00+06:00"}
Expected:
  HTTP 201
  start_time is "2026-12-02T06:00:00Z" or "2026-12-02T06:00:00+00:00"
  end_time is "2026-12-02T07:00:00Z" or "2026-12-02T07:00:00+00:00"

T-WIN-ZERO - zero duration rejected
Body:
  {"room_id":ROOM_A_ID,"start_time":"2026-12-03T09:00:00Z","end_time":"2026-12-03T09:00:00Z"}
Expected:
  HTTP 400
  JSON has code:"INVALID_BOOKING_WINDOW"

T-WIN-NEG - negative duration rejected
Body:
  {"room_id":ROOM_A_ID,"start_time":"2026-12-03T10:00:00Z","end_time":"2026-12-03T09:00:00Z"}
Expected:
  HTTP 400
  JSON has code:"INVALID_BOOKING_WINDOW"

T-WIN-9H - duration over 8h rejected
Body:
  {"room_id":ROOM_A_ID,"start_time":"2026-12-03T09:00:00Z","end_time":"2026-12-03T18:00:00Z"}
Expected:
  HTTP 400
  JSON has code:"INVALID_BOOKING_WINDOW"

T-WIN-HALF - non-whole-hour duration rejected
Body:
  {"room_id":ROOM_A_ID,"start_time":"2026-12-03T09:00:00Z","end_time":"2026-12-03T10:30:00Z"}
Expected:
  HTTP 400
  JSON has code:"INVALID_BOOKING_WINDOW"

T-WIN-PAST - past start rejected with no grace
Body:
  {"room_id":ROOM_A_ID,"start_time":"2026-01-01T09:00:00Z","end_time":"2026-01-01T10:00:00Z"}
Expected:
  HTTP 400
  JSON has code:"INVALID_BOOKING_WINDOW"

T-CONF-OVERLAP - overlapping booking rejected
Precondition:
  MEMBER_BOOKING_ID exists for 2026-12-01T09:00:00Z to 11:00:00Z.
Body:
  {"room_id":ROOM_A_ID,"start_time":"2026-12-01T10:00:00Z","end_time":"2026-12-01T12:00:00Z"}
Expected:
  HTTP 409
  JSON has code:"ROOM_CONFLICT"

T-CONF-BACK2BACK - adjacent booking allowed
Body:
  {"room_id":ROOM_A_ID,"start_time":"2026-12-01T11:00:00Z","end_time":"2026-12-01T12:00:00Z"}
Expected:
  HTTP 201
  status:"confirmed"

T-BOOK-ROOM-NOTFOUND - foreign org room behaves as not found
Authorize with MEMBER_A_ACCESS.
Body:
  {"room_id":ROOM_B_ID,"start_time":"2026-12-04T09:00:00Z","end_time":"2026-12-04T10:00:00Z"}
Expected:
  HTTP 404
  JSON has code:"ROOM_NOT_FOUND"

T-REF-FORMAT - booking reference code format
Any successful booking response should include:
  reference_code matching format CW-000001, CW-000002, etc.

T-QUOTA - fourth booking in next 24h is rejected
Action:
  Create 3 confirmed bookings for MEMBER_A_ACCESS with start_time inside the next 24 hours from now, each in a different free slot.
  Try a 4th booking also inside the next 24 hours.
Expected for 4th booking:
  HTTP 409
  JSON has code:"QUOTA_EXCEEDED"

T-CONF-CONC - no double booking under concurrent attempts
Manual browser approximation:
  Open two Swagger tabs, authorize both with the same token, prepare the same POST /bookings body for the same room/time, submit as close together as possible.
Expected invariant:
  Exactly one request returns HTTP 201.
  The other returns HTTP 409 with code:"ROOM_CONFLICT".

  POST /bookings/{id}/cancel

High-value final sanity sequence
--------------------------------
Run these in this order if time is short:
1. GET /health -> 200 {"status":"ok"}
2. Register admin/member and login both -> expected roles/tokens.
3. Duplicate register bob -> 409 USERNAME_TAKEN.
4. Member POST /rooms -> 403 FORBIDDEN.
5. Admin create room -> 201; member create 2h booking -> 201 price_cents 2000.
6. Overlap same room/time -> 409 ROOM_CONFLICT.
7. Back-to-back booking -> 201.
8. Offset datetime +06:00 -> stored/echoed in UTC.
9. Zero/negative/past booking windows -> 400 INVALID_BOOKING_WINDOW.
10. Member GET /bookings only sees own bookings.
11. Other member/admin booking detail as bob -> 404 BOOKING_NOT_FOUND.
12. Cancel >48h booking -> 100% refund; re-cancel -> 409 ALREADY_CANCELLED.
13. Availability removes cancelled booking immediately.
14. Stats/report update after create/cancel and exclude cancelled bookings.
15. Export header is exact and cross-org room export returns 404 ROOM_NOT_FOUND.
